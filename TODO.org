Tracks high-level features not yet implemented. Each entry is a goal

* DONE Split cron-sync configuration out of the tracked script

Real design flaw: ~patchwork-cron-sync.el~ is tracked by git, but its
own Commentary/README instructed users to edit its ~(setq
patchwork-servers ...)~ block directly to configure it. Any user who
did that now has an uncommittable local change to a tracked file --
every future `git pull` touching that file's functional logic (which
this whole session did repeatedly: --list-servers, single-server
mode) either silently never reaches them or conflicts outright with
their own settings, exactly the "mixing functional aspects with
configuration" problem.

Fixed by splitting the two apart properly: ~patchwork-cron-sync.el~
now contains only functional logic, no configuration of its own, and
loads a sibling ~patchwork-cron-config.el~ for `patchwork-servers` and
friends -- a file that is NOT tracked by git (added to `.gitignore`).
A new ~patchwork-cron-config.el.example~ (tracked, a template only,
never itself loaded/executed) is what a user copies to the real,
gitignored path and edits. The real config's location can also be
pointed elsewhere entirely via `$PATCHWORK_CRON_CONFIG` (e.g. a private
dotfiles repo), rather than assuming it always lives next to the
script. A missing config file errors with a clear, actionable message
rather than silently syncing zero servers.

Verified: missing-config-file case errors with the expected message;
a real split config (via $PATCHWORK_CRON_CONFIG, pointed at two live
mock-server instances) works correctly across all three invocation
modes (plain, --list-servers, single-server-argument) and the parallel
wrapper script end-to-end, exactly as before the split.

Follow-up, same problem from a different angle: a hand-copied template
still has to be kept in sync by hand every time the interactive config
changes, which is the same class of drift risk one level down. Added
~patchwork-generate-cron-config.el~, a pure-elisp script (mirrors
`patchwork-cron-sync.el`'s own `command-line-args-left` handling) that
loads a given init file (defaulting to `user-init-file`) and dumps the
current value of every sync-relevant `patchwork-*` variable (an
explicit fixed list -- deliberately excludes UI-only settings like
highlighting/buffer names that don't affect what a sync does) into
`patchwork-cron-config.el` as plain `(setq VAR 'VALUE)` forms. Reuses
`$PATCHWORK_CRON_CONFIG` symmetrically as the *output* location, same
variable `patchwork-cron-sync.el` reads its config location from.
Unlike the sync script, this one's whole point is to load the user's
real init file -- an acceptable cost for a one-off run by hand, not a
five-minute cron tick.

Verified: round-trips correctly (a fake init.el with per-server
`:since-format`/`:sync-timeout`/`:events-omit-project` overrides and a
customized `patchwork-sync-lookback-days`, loaded back by
`patchwork-cron-sync.el`, produces the identical values); UI-only
variables set in the same fake init.el are correctly excluded; an init
file that errors partway through keeps whatever was set *before* the
error (not silently reverting to defaults) with a clear warning rather
than failing outright; a lazily-loaded patchwel setup (nothing actually
binds `patchwork-servers`) produces a warning instead of a silently
empty config.

* DONE Compose replies via gnus-msg-mail when Gnus is running

`patchwork-mail-compose-hook' (added earlier so a hand-added Gcc header
could be inserted) turned out not to be enough on its own: a Gcc header
does nothing unless something is actually watching for it at send time,
and that watcher only gets wired up by Gnus's own composition setup
(`gnus-setup-message'), which a plain `message-mail' call never goes
through.

New `patchwork-mail--compose' prefers `gnus-msg-mail' -- which is,
per its own docstring, "like `message-mail', but with Gnus
paraphernalia, particularly the Gcc: header for archiving purposes" --
whenever Gnus is actually running in the session (`(and (featurep
'gnus) (fboundp 'gnus-msg-mail) (gnus-alive-p))`), falling back to
plain `message-mail' otherwise (`gnus-msg-mail' has this same fallback
built in too, but the checks here avoid ever autoloading/loading Gnus
just to ask, so there's no cost or behavior change for anyone not
already using it).

Confirmed via `(find-function-noselect 'gnus-msg-mail)` that it wraps
the same `message-mail' call inside `gnus-setup-message', and
empirically (a minimal faked-"alive" Gnus session) that composing this
way does invoke real Gnus message-composition machinery
(`gnus-configure-posting-styles', `gnus-message-citation-mode', etc.)
that a plain `message-mail' buffer never triggers -- confirming this is
the actual missing mechanism, not just a documentation gap. Also found
along the way: a plain `let'-binding of the special variable `features'
does not fool `featurep' (some internal caching bypasses it), so the
new tests stub `featurep' itself via `cl-letf' rather than trying to
fake feature-loadedness that way.

* DONE Parallel cron sync, one process per server

`patchwork-cron-sync.el' now supports a per-server mode: given a
server URL as its argument (everything after Emacs's own `--' on the
command line), it syncs only that one server rather than every
configured one. A new `patchwork-cron-sync-parallel.sh' drives this
once per configured server concurrently instead of one process working
through all of them serially -- a sync's wall-clock time is dominated
by serialized network round-trips, not by Emacs Lisp itself, so this
is a real speedup for multi-server setups without needing to rewrite
anything in another language. The server list is read from
`patchwork-cron-sync.el' itself via a `--list-servers' mode (prints
each configured `:url', no sync, no db access), so it's never
duplicated between the two files.

Two real bugs found and fixed while implementing this, both re: Emacs
batch argument handling, neither obvious in advance:

- `command-line-args-left' still contains the literal "--" marker at
  the point a `-l' script runs (Emacs only treats it as "stop parsing
  my own options" when its own sequential loop reaches that position,
  not before). Naively filtering `command-line-args-left' by deleting
  "--" *from that same global* corrupts Emacs's own remaining-argument
  queue, since it's the actual mutable state Emacs's top-level loop
  keeps popping from after the script returns, not just a read-only
  snapshot -- once the script returns, Emacs resumes past where "--"
  used to be and tries to parse whatever's left (e.g.
  "--list-servers") as one of its own unrecognized options. Fixed by
  reading a locally-filtered copy for the script's own use and setting
  the global to nil when done, so Emacs's own loop has nothing left to
  trip over.
- A non-`-Q' `emacs` invocation can load site-wide init files
  (`/usr/share/emacs/site-lisp/site-start.d/*`, common on Linux
  distro packages) that print to stdout -- harmless noise for a plain
  sync, but fatal for `--list-servers', whose output the wrapper
  script parses as a literal list of server URLs. Fixed by invoking
  Emacs with `-Q' in the wrapper script.

Verified against two real mock-server instances standing in for two
configured servers: `--list-servers` prints exactly the two URLs;
single-server-argument mode touches only its own server's rows in the
shared db; two such processes running concurrently both complete and
write cleanly to the same WAL-mode db with no locking errors; the
no-argument (serial, unchanged-default) mode still works exactly as
before.

* DONE tmt-based test suite with a mock Patchwork server and CI

74 ~ert-deftest~ tests across 9 areas (~tests/test-db.el~,
~-api.el~, ~-cache.el~, ~-git-apply.el~, ~-git-review.el~,
~-git-prune.el~, ~-ui-highlight.el~, ~-ui-render.el~,
~-mail.el~), run individually via ~tests/run-ert.sh tests/test-X.el~
or as a whole via ~tmt~ (~tmt run --all provision --how local
--feeling-safe plan --name /plans~; newer ~tmt~ requires
~--feeling-safe~ for the ~local~ provisioner), wired into
~.github/workflows/test.yml~ for CI (installs Emacs via
~purcell/setup-emacs~, ~pip install 'tmt[report-junit]'~, runs the
plan, reports JUnit results as an artifact).

~tests/mock_patchwork_server.py~ is a stdlib-only Python
~ThreadingHTTPServer~ serving Patchwork-shaped JSON fixtures (3
series/4 patches/comments/checks/events/covers), a real per-patch mbox
(via ~git format-patch~ against a scratch repo built at startup, not
hand-written text -- each patch adds its own distinct file so any
subset/order applies cleanly with no inter-patch dependency), an
Anubis-style HTML challenge page toggleable per path, and a
~/_control/*~ mutation API (reset, since= format gating, forced
per-path HTTP status/delay, patch-state transitions, challenge
toggling, a request log) that ERT tests drive to exercise scenarios
against one shared long-lived server process per test file (reset via
~/_control/reset~ at the top of `patchwork-test-with-mock-server' gives
each ~ert-deftest~ fixture-like isolation without ERT's native
support for it). ~tests/test-helpers.el~ also provides
~patchwork-test-with-temp-db~ (isolates ~patchwork-db--connection~'s
module-global cache per test) and ~patchwork-test-with-temp-repo~/
~patchwork-test-make-patch-fixture~ (real scratch git repos and
~git format-patch~-generated mboxes for the apply/am/branch-strategy
tests).

Writing this actually caught two real, previously-unexercised bugs in
shipped code: ~patchwork-apply-patch~'s default args
(~"--3way" "--reject"~) can't both be passed to ~git apply~ at once
(fixed to just ~"--reject"~, matching what its reject-file handling
already assumed); fixture dates hardcoded to a fixed calendar date
would have silently aged out of the default 30-day sync lookback
window depending on when the suite runs (fixed to be computed relative
to "now" at fixture-build time).

Deliberately verified the pass/fail wiring itself, not just that tests
run: temporarily broke one assertion, confirmed ~tmt run~ exits
non-zero and reports "N passed and 1 failed" rather than silently
staying green, then reverted.

* DONE Prune cached patch mbox files

`patchwork-git-download-patch' caches every patch mbox it ever
downloads in `patchwork-git-temp-dir' indefinitely -- nothing cleaned
up after itself, so the directory only grows. Two ways to reclaim
that space:

- ~M-x patchwork-git-prune-patches~ deletes cached files older than
  `patchwork-patch-cache-max-age-days' (default 30), or with a prefix
  arg, every cached file regardless of age. Prompts for confirmation,
  reports how many files/how much space were freed.
- Automatic, during sync: `patchwork-prune-on-terminal-states' (a list
  of state strings, defaulting to accepted/rejected/deferred/
  changes-requested/superseded/not-applicable -- on by default,
  adjustable since state names vary by deployment, or set to nil to
  disable) drives `patchwork-cache--maybe-prune-patch', called from
  `patchwork-cache--sync-patch'. When a sync observes a patch's state
  transitioning from something not in that list into something that
  is, its cached mbox (if it has one) is deleted immediately --
  review/apply work is presumably done by then. Keyed off a real
  state *transition* (comparing the previously-cached row's state to
  the freshly synced one), not just "is this patch's state terminal",
  so re-syncing an already-terminal patch is a silent no-op rather
  than repeatedly trying to delete an already-gone file.

Verified: the manual prune command correctly ages files by mtime
(leaves recent/non-.patch files alone) and honors declining the
confirmation prompt; the sync-time hook correctly leaves the mbox
alone on a non-terminal-to-non-terminal transition, prunes it on a
non-terminal-to-terminal one, and is a no-op re-syncing an
already-terminal patch or with the state list set to nil.

* TODO Asynchronous/offline-tolerant patch delegation

Setting a patch's delegate (~patchwork-api-set-delegate~) is currently
a direct, synchronous API call: if the server is unreachable, the
request just fails. Instead:

- When a delegate change is requested but the server call fails
  (timeout, connection error, etc.), save the intended change locally
  (patch id, server, desired delegate, and the delegate value we
  observed at request time) rather than discarding it.
- On each subsequent successful sync/connection to that server, retry
  any pending delegate changes.
- Before applying a queued change, re-check the patch's *current*
  delegate against what we observed when the change was queued. If it
  has changed in the meantime (someone else delegated it, or changed
  it another way), do not blindly overwrite — the change is stale and
  needs to surface as a conflict rather than applying silently.
- Needs a small local queue (a new sqlite table, most likely) and a
  place to report queued/failed/conflicting changes back to the user.

* DONE Per-series sync from the series detail buffer

~M-x patchwork-fetch-series~ (~G~ in the main listing buffer) prompts
for a server and a series id, fetches and caches that one series --
its patches, comments, and checks, same as a routine sync -- via the
new ~patchwork-cache-sync-series~, regardless of
~patchwork-sync-lookback-days~ or which project it belongs to (a
series' own data already says that, so no project prompt is needed),
then opens its detail buffer. Verified against a real ~2023 series on
patchwork.ozlabs.org, over two years outside any normal sync window,
fetched cleanly with its patch, all three comments, and check counts.

* TODO Threaded comment view across a series' patches

The series detail buffer currently lists each patch's comments as a
flat, chronological list per patch. Instead, render all comments
across every patch in the series in a threaded fashion (reply
relationships, not just date order) so a review conversation reads the
way it would in a mail client or code review tool. Confirmed (while
implementing comment replies) that comments do carry real In-Reply-To
and References headers via the API, cached in the `comments' table's
~in_reply_to_header~/~references_header~ columns -- so threading can
be built from real header data, not ordering/quoting heuristics.

* DONE Apply a series and start a review

~R~ (~patchwork-series-review-at-point~ in the main listing,
~patchwork-series-detail-review~ in a series detail buffer, or
~M-x patchwork-review-series~) applies every patch in a series to its
project's configured git tree and opens a browsable view of the
result, via two new pieces in ~patchwel-git.el~:

- ~patchwork-apply-series-as-commits~ downloads each patch's mbox and
  hands the whole set to a single ~git am --3way~ call, so every patch
  becomes its own real commit (author, date, and commit message
  preserved from the mail), unlike the existing
  ~patchwork-apply-series~/~git apply~ path, which only ever touches
  the working tree. ~--3way~ means a failing patch falls back to a
  real merge with conflict markers rather than simply refusing to
  apply. Returns the before/after HEAD commit hashes, or nil (leaving
  the tree mid-~am~) if a patch fails to apply -- in which case a
  ~*patchwork-am: <repo>*~ buffer is shown automatically with ~git
  am~'s output, the patch it was blocked on, and each conflicted file
  as a button to open directly, so there's no need to drop to a shell
  to see what happened. ~patchwork-git-am-status~ shows the same thing
  on demand (e.g. coming back to it later); ~patchwork-git-am-continue~/
  ~-skip~/~-abort~ run the matching ~git am~ subcommand and refresh
  that buffer with the result, all from within Emacs.
- ~patchwork-review-backends~ is a pluggable hook
  (~run-hook-with-args-until-success~, called with REPO-DIR BEFORE-REV
  AFTER-REV) of "start a review for these applied commits" backends,
  tried in order. Built-in backends: ~patchwork-review-backend-magit~
  (magit's log buffer over the range, when magit is loaded) and
  ~patchwork-review-backend-vc~ (a ~log-view-mode~ buffer via
  ~vc-print-log-internal~ with the Git backend passed explicitly,
  since the usual ~vc-print-root-log~/~vc-deduce-backend~ path infers
  the backend from the current buffer's major mode/~vc-mode~ and fails
  right after applying a series, before any file-visiting buffer under
  version control exists). Both give the usual n/p-between-commits,
  RET-to-view-diff navigation their mode already provides.
  ~git-timemachine~ doesn't fit this hook (it walks one file's history,
  not a range of commits across a tree) but still works as normal on
  any file touched by the series once applied.

Per-project git tree configuration: ~patchwork-project-git-trees~ (an
alist keyed by (SERVER-URL . PROJECT-SLUG)) plus
~patchwork-project-git-tree~, which looks up a project's configured
tree or prompts once and offers to save the answer, since different
projects usually need different trees.

Per-project branch-apply strategy: ~patchwork-project-branch-strategies~
(same (SERVER-URL . PROJECT-SLUG) alist shape, falling back to
~patchwork-default-branch-strategy~) controls whether applying a
series creates its own branch rather than landing on whatever is
currently checked out. nil (the default) means apply as-is; a string
of the form ~"BASE:TARGET-TEMPLATE"~, e.g. ~"upstream/main:review_%i"~,
means create and check out a new branch named by substituting the
series id for any ~%i~ in TARGET-TEMPLATE (~review_514346~ here), based
on BASE (~upstream/main~). Parsed by ~patchwork-parse-branch-strategy~,
applied by ~patchwork-git--maybe-checkout-branch~ (used by both
~patchwork-apply-series~ and ~patchwork-apply-series-as-commits~, so it
applies to reviews and plain working-tree-only applies alike). Errors
out rather than reusing or resetting an already-existing target
branch -- applying the same series twice under a strategy is a
mistake to surface, not silently overwrite.

Verified against a scratch repo and hand-built two-patch mbox series
(both the happy path -- two real commits produced with correct
author/date/message, and the vc/magit backends each opening the
correct range in their respective log views -- and the conflict path,
where ~git am~ fails cleanly and leaves the repo in a normal mid-~am~
state for `git am --continue`/`--abort`).

* DONE Reply to a comment as a mail message

From a comment shown in a series detail buffer, ~r~
(~patchwork-series-detail-reply-to-comment~) spawns a wide-reply
~message-mode~ buffer, implemented in the new ~patchwel-mail.el~:
To is the comment author's email; Cc is the original To/Cc addresses
minus the new To and minus ~user-mail-address~; Subject gets a "Re: "
prefix unless already present; In-Reply-To is the comment's Message-ID;
References is the comment's own References chain plus its Message-ID
appended. The comment body is quoted one level deeper than it already
was. Required extending the `comments' table with ~msgid~, ~subject~,
~submitter_email~, ~to_header~, ~cc_header~, ~references_header~, and
~in_reply_to_header~ columns (schema version bumped to 3), populated
from the real Patchwork comment API, which does expose full mail
headers per comment.

Note: References/In-Reply-To can't just be passed via
`message-mail's OTHER-HEADERS -- message-setup-1 forcibly regenerates
both from `message-reply-headers' (unset here, since there's no real
Gnus reply buffer behind this), silently dropping whatever was
supplied. Add them afterward with ~message-add-header~ instead, which
only inserts a header if not already present.

* DONE Expand-all/collapse-all in the series detail buffer

~+~/~-~ (~patchwork-series-detail-expand-all~/
~patchwork-series-detail-collapse-all~) now expand/collapse every
patch and comment in the current series detail buffer at once,
mirroring the main listing buffer's group-level ~+~/~-~. Also added,
same session: ~n~/~p~ move to the next/previous patch's summary line,
~N~/~P~ to the next/previous comment's, regardless of expand state
\(the id properties live only on the summary line itself, never on the
expanded content beneath it, so navigation is just a line-by-line walk
checking for the right text property).
